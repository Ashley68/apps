Stuff App Create-Read-Update-Delete Application Template

Web Applications generally involve using a web browser to interact with data in a database via a web browser. Interact means to Create, Read, Update, or Delete data. Data can be articles, blog posts, classified ads, catalogs, personal profiles, pictures, or any other kind of data.

In this tutorial we will create a general purpose Create-Read-Update-Delete type application that we'll call Stuff. The app will create a database table where you can create your stuff, pull stuff to view or Read, Update your stuff and delete stuff. Once you understand this general purpose app you can use it as a template for other apps to modify and build upon. And we'll create more tutorials that do just that.

We will assume you already have a Django project started with the initial files set up. You can do that by following the simplesite tutorial to create a simple two page website using the Django framework.

Prerequisites: This tutorial assumes you are familiar with HTML and CSS; a basic command line interface (e.g., Windows Command Prompt, OSX Terminal, Linux Bash); the basics of the Python Programming Language; and preferrably you have already gone through the Django "Polls App" Tutorial at Djangoproject.org. At a minimum, you should have the following already installed: A text editor, Python version 2.7 or above, and Django.

Setup

  • Open your command line interface (e.g., Terminal for Mac, Command Prompt for Windows, Bash for Linux).
  • Change Directories to your site's upper directory cd /localsites/simplesite or whatever the path is to the manage.py file.
  • Create the folder and files for the application that we'll call stuff. In the command line enter python manage.py startapp stuff
  • A new folder named stuff will be created with some files in it (e.g., views.py, models.py).
  • Open the settings.py file. Insert a new row under INSTALLED_APPS = ( and enter 'stuff',. Now Django will know to include this app with your site.

Create the Database Model

  • We will start by populating the model.py file. This is where we define the database table(s) and fields.
  • Open the models.py file in the stuff directory and populate it with the following: 
    from django.db import models

class Thing(models.Model):  
    title = models.CharField(max_length=255)
    slug = models.SlugField(max_length=255)
    description = models.TextField()
    python_developer = models.BooleanField()

    def __unicode__(self):
        return self.title
    This will do the following:
  • Import the models module from django located at the following path (starting from your Python or Virtualenv folder) Lib/site-packages/django/db/models.
  • Create a class called Thing. When you sync your Django files with your database (explained later) this will create a table called Thing. Each one of the attributes (title, type_of_thing, slug, etc.) will become a field in the table.
  • You must assign each field to a field type (e.g., CharField, TextField, BooleanField). Django converts these to the database field type. CharField, and SlugField require that you specify the maximum length.
  • Django automatically generates a field called id, assigns it an auto generated integer starting with 1, and makes "id" the primary key.
  • You can also populate it with the below more complicated version. 
    from django.db import models

class Thing(models.Model):  #This will create a table in the DB with the class name as table name
    title = models.CharField(max_length=255)
    THING_CHOICES = (
        ('blog', 'Blog'),
        ('classifieds', 'Classified Ads'),
        ('catalog', 'Catalog'),
        ('photos', 'Photos'),
        ('survey', 'Survey'),
    )
    type_of_thing = models.CharField(max_length=50, choices=THING_CHOICES)
    slug = models.SlugField(unique=True, max_length=255, blank=True)
    description = models.TextField()
    python_developer = models.BooleanField(default=True)
    creation_date = models.DateTimeField(auto_now_add=True)

    def __unicode__(self):
        return self.title

    class Meta:
        ordering = ['-title']
  • THING_CHOICES is the name we gave to a constant that is called by the choices argument. Django converts these to choices available in a select box. They are in pairs. The item on the left is what goes in the database, on the right is what appears in the dropdown box.
  • We also added several optional arguments to fields.
  • Creation_date will be automatically entered when the form is submitted and a record is created in the database.
  • At the end we added a class to sort the results alphabetically by title when they are displayed. Because we started it with a minus sign it will go in reverse order (z to a).

Forms

  • Now we will create the form that will be displayed to users to create and update records in our database.
  • One of Django's core principles is Don't Repeat Yourself (DRY). If you leverage Django's functionality then you don't need to rewrite the fields in your form, rather you pull in the fields from the models.py file.
  • Create a file called forms.py, save it to the stuff directory, and populate it with the following: 
    from django import forms
from models import Thing

class StuffForm(forms.ModelForm):

    class Meta:
        model = Thing
        # uncomment the below line to include only specific fields.
        # fields = ('title', 'description', 'python_developer')s
  • This imports Django's forms module, and it imports the Thing class we just created in our models.py file.
  • We create a class for the form that we're calling StuffForm. It converts the attributes in the Thing class into fields on the form so we don't have to reenter them.
  • Django excludes automatically generated fields from forms such as the "id" field and date created.
  • If we don't want all the fields in the Thing class to appear on our form then we can specifically exclude them. Right below model = Thing we would insert exclude = ('slug'). Or, conversely, we can list only the fields we want fields = ('title', 'description', 'python_developer').

Templates

  • Now the form is created. Let's make the html files to 1) Display a list of data in the database (our stuff); 2) Display the details of individual things; 3) A form to create new things; 4) A form to update existing things.
  • We will assume your site already has a templates directory and a base.html file with the navigation bar. Open the base.html file and in the nav element add a navigation menu link to the stuff app <a href="/stuff">Stuff</a>
  • Within the templates directory, create another directory called "stuff".
  • Create a file called index.html, save it to the templates/stuff folder, and populate it with the following: 
    {% extends 'base.html' %}
{% block title %}List Stuff{% endblock %}
{% block content %}

<h2>List of Stuff</h2>

    <p><a href="{% url "create_stuff" %}">Add New Stuff</a></p>
    {% for thing in stuff %}
        <a href="{% url "thing_details" thing.id %}">{{ thing.title }}</a><br/>
    {% endfor %}

{% endblock %}
    • When this html page is opened it will have a link to "Add New Stuff" and below that it will list all the records we have in the database. It uses a for loop to pull all the data in.
    • Within the For Loop we are adding a link for each record that goes to the thing_details template with the thing.id appended to it. What is displayed is the title field for each record.
  • Create a file called thing_details.html, save it to the templates/stuff folder, and populate it with the following: 
    {% extends 'base.html' %}
{% block title %}Details{% endblock %}
{% block content %}

  <h2>{{ stuff.title }}</h2>
  <p><a href="{% url "edit_stuff" thing.id %}">Edit your stuff</a></p>
  <p><b>Created:</b> {{ thing.creation_date|date:"D, Y/m/d" }}</p>
  <p>
  {% if thing.description %}
    {{ thing.description }}
  {% else %}
    You got no stuff :(
  {% endif %}
  </p>

{% endblock %}
    • This page shows the details of a specified individual record from your database. The database automatically generates a sequential id number for every record. The record is called when the URL is appended by stuff and the record's id number like so /stuff/1
    • At the top of the page we are displaying the title field for the record.
    • Below that is a link to the "edit_stuff" page where you can edit the record. That is called by appending the url with /stuff/edit/id number
    • In this case we are including the creation date. That's relevant for things like blog posts, articles, pictures, but you can delete that line for things like catalogs.
    • The next line is an if statement. If the description field is not empty then show the description field, otherwise show some message.
  • Create a file called create_stuff.html, save it to the templates/stuff folder, and populate it with the following: 
    {% extends "base.html" %}
{% block title %}Create New Stuff{% endblock %}
{% block content %}
    <h2>Create New Stuff</h2>
      <form action="" method="post">{% csrf_token %}
        {% for field in form %}
            <div class="field_container">
                {{ field.errors }}
                {{ field.label_tag }}</br>
                {{ field }}
            </div>
        {% endfor %}
      <input type="submit" value="Submit" />
      </form>
{% endblock %}
    • We are displaying a form for creating a new record and inserting it into the database. It uses the post method.
    • The {% csrf_token %} is a security feature provided by Django's form module to prevent cross site request forgeries where data is inserted in your database bypassing the form.
    • A For Loop is used to generate the fields from the form.py file (which calls the fields from the models.py file). We are including a variable for the field error messages, the label tag and an input field to enter the data.
  • Create a file called edit_stuff.html, save it to the templates/stuff folder, and populate it with the following: 
    {% extends "base.html" %}
{% block title %}Edit Stuff{% endblock %}
{% block content %}
  <form action="" method="post">{% csrf_token %}
    <h2>Edit Your Stuff</h2>
    {% for field in form %}
        <div class="field_container">
            {{ field.errors }}
            {{ field.label_tag }}</br>
            {{ field }}
        </div>
    {% endfor %}
  <input type="submit" value="Submit" />
  </form>
{% endblock %}
    • This form is essentially the same as the create_stuff form except it calls an existing record in the database and allows you to modify the values in the fields.

URLs

  • Now we will dispatch the URLs. Create a file called urls.py, save it to the stuff folder, and populate it with the following: 
    from django.conf.urls import patterns, url

urlpatterns = patterns('',
    url(r'^$', 'stuff.views.index', name="stuff"),
    url(r'^(?P\d+)/$',
         'stuff.views.thing_details',
          name='thing_details'),
    url(r'^create/$',
         'stuff.views.create_stuff',
          name="create_stuff"),
    url(r'^edit/(?P\d+)/$',
         'stuff.views.edit_stuff',
         name='edit_stuff'),
  • This file imports the url and patterns functions from Django's urls module located at (from your Python or virualenvs folder) Lib/site-packages/django/conf/urls
  • The patterns function looks for a match with the urls inside it when a URL is entered. The url function matches a URL with a function. It also allows you to give the URL a name.
  • You may notice that /stuff/ is not part of these URLs. That's because we put that in the site's main urls.py file.
  • Open the main urls.py file. It's in the sites lower directory, where the settings.py file is.
  • We need to connect the urls file we just created with the site's main urls.py file. Below urlpatterns = patterns('', insert the following line: 
        (r'^stuff/', include('stuff.urls')),
    This uses the include function to include the urls.py file from the stuff folder whenever a url is appended with "stuff/".
  • Note, we are using the option to give the urls names. That allows us to use the link name rather than the actual url when we put links throughout our site. That way if we change the URLs we can do it centrally in the URLs file and not have to change the links in the html files.

Views

  • Now we will create the functions to display the pages and process the forms.
  • Open the views.py file in the stuff directory and populate it with the following: 
    from django.shortcuts import get_object_or_404, render_to_response, redirect
from django.template import RequestContext
from stuff.models import Thing
from stuff.forms import StuffForm

def index(request):
    stuff = Thing.objects.all()
    return render_to_response('stuff/index.html',
                                      {'stuff': stuff},
                      context_instance=RequestContext(request))

def thing_details(request, thing_id):
    thing = get_object_or_404(Thing, pk=thing_id)
    return render_to_response('stuff/thing_details.html',
                                                  {'thing': thing},
                                context_instance=RequestContext(request))

def create_stuff(request):
    if request.method == 'POST':
        form = StuffForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect('stuff')
    else:
        form = StuffForm()
    return render_to_response('stuff/create_stuff.html',
                                                          {'form': form},
                                context_instance=RequestContext(request))

def edit_stuff(request, thing_id):
    thing = get_object_or_404(Thing, pk=thing_id)
    if request.method == 'POST':
        form = StuffForm(request.POST, instance=thing)
        if form.is_valid():
            form.save()
            return redirect('detail_stuff', thing_id)
    else:
        form = StuffForm(instance=thing)
    return render_to_response('stuff/edit_stuff.html',
                                                          {'form': form},
                                context_instance=RequestContext(request))
  • The index view (index function):
    • At the top of the file we imported the Thing class from our models.py file. In the function we created a variable called "stuff" and populated it with all the objects in the Thing table in our database.
    • At the top of the file we also imported the render_to_response function from the Django Shortcuts module. This function will load the page we include as the first argument 'stuff/index.html',.
    • We are also passing the variable we just defined to the page we are rendering {'stuff': stuff},.
    • At the top of the file we imported the context_instance=RequestContext(request) function from the Django template module. Among other things, this function will process the csrf token.
  • The thing_details view (thing_details function):
    • This function includes the thing id as an argument. A variable called thing is created from the Thing table in the database based on the id number. The get_object_or_404 function will give a 404 page not found error if the id number is not in the database.
    • The render_to_response function renders the thing_details file and passes the "thing" variable we just created.
  • Create_stuff view:
    • The create_stuff template displays a template to create a new "thing" and add it to the "Thing" table in your "stuff" database.
    • The create_stuff view does two things. It renders create_stuff.html template with an empty form to fill out. It also processes the form when it is submitted. It uses an if statement to know which action to take.
    • If the form is filled out it will be submitted using the "post" method, making if request.method == 'POST': True.
    • At the top of the file we imported the StuffForm class from the forms.py file. Below the if statment we created a variable named "form" and populated it with the POST data that was submitted.
    • Django's form module does some validation to ensure the data entered is safe from sql injections. If so it saves the data and redirects you to the stuff URL using the imported redirect function imported from Django shortcuts.
    • If the if statement was False (i.e., there was no POST data) then the else statement is executed. A variable named "form" is created and populated with the StuffForm class imported at the top of the file from the forms.py file. Then the create_stuff.html file is rendered and the form variable is passed to it.
  • Edit_stuff view (edit_stuff function):
    • The edit_stuff view is mostly the same as the create_stuff view except it starts with an existing record rather than a blank record.
    • the function is called when a URL has a number following stuff/edit/ (see the urls.py file). It takes the id number as an argument to the function.
    • You create a variable called "thing" and populate it with the database record, or if the id number doesn't exist it renders a 404 url not found error message.
    • The object data in the "thing" variable is passed to the form when the page is rendered.

Activate the Admin Console

  • As the site administrator, you can use the Admin Console to create, read, update, or delete stuff.
  • Activate Django's Admin App.
    • Open the settings.py file. Under INSTALLED_APPS uncomment the Admin App line 'django.contrib.admin',.
    • Now sync the database. Open the command line interface and change directories to the upper directory of your site (where manage.py is). Type python manage.py syncdb. This will create tables in your database for both the admin app and the stuff app.
    • The first time you run syncdb for the site you are prompted for a username and password which will make you the superuser.
    • Edit the main urls.py file (in the same directory as settings.py) and uncomment the three lines that reference admin so it looks like the below (excluding other links or code you may have). 
      from django.conf.urls import patterns, include, url

from django.contrib import admin
admin.autodiscover()

urlpatterns = patterns('',
    url(r'^admin/', include(admin.site.urls)),
)
  • The admin console and the stuff app are now activated. Let's now add the ability to manage the stuff app through the Admin Console. Create a file named admin.py, save it to the stuff folder, and populate it with the following: 
    from django.contrib import admin
from stuff.models import Thing

admin.site.register(Thing)
  • Okay, we are done. Now let's test it.

Testing

  • Run the virtual server and test the app in your browser.
  • Open the command line interface.
  • from the upper site directory (the folder with the manage.py file in it) enter python manage.py runserver
  • Go to the browser and put in your local ip address and port 8000 as url: http://127.0.0.1:8000/
  • Click on the navigation bar's "stuff" link. It should take you to the index.html page listing any stuff you have... which is none at this point.
  • Click on the create stuff link. It should take you to the create_stuff form.
  • Fill out the form both incorrectly, to see the error messages, and correctly.
  • When filled out correctly you should be directed back to the stuff page. You should see the item you just entered.
  • Click on the item and it should take you to the thing_detail page. Click on the Edit My Stuff link.
  • This should take you to the edit_stuff page with the form. Fill out the form both incorrectly, to see the error messages, and correctly.
  • If it all works, you are done!
  • Once you are comfortable with this app and how it works, you can modify it to a different kind of app that Creates, Reads, Updates, and Deletes data, like a blog, articles, surveys, catalog, photos, profiles, etc.
